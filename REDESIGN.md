# WhichCity Redesign 指导文件

> 编码规范仍遵循 [RULES.md](RULES.md)，数据质量仍遵循 `.github/instructions/data-quality.instructions.md`。
> 本文件补充的是**产品方向、信息架构、交互思路**。
>
> **本文件分为三部分**：
> - **第一部分：硬约束**（必须遵守，违反即 bug）
> - **第二部分：设计原则**（方向性指导，改动时对照思考）
> - **第三部分：参考方案**（具体措施的素材库，仅供参考，不是规范）

# 第一部分：硬约束

以下条目是工程和产品的底线，任何改动都不得违反。

### 1. URL 兼容

- 现有 URL 结构 `/[locale]/city/[slug]`、`/[locale]/ranking`、`/[locale]/compare/[pair]` **不变**
- 老链接不会 404

### 2. 数据完整性

- cities.json 的 50+ 字段**只增不删**
- 前端可以选择不展示某些字段，但不得从数据源移除

### 3. SEO 基础设施

- OG images、sitemap、JSON-LD、metadata **保留**
- 城市详情页的核心数据（收入、住房、工时等）保持**默认可见**（不折叠），折叠内容的 SEO 权重低于可见内容

### 4. SSR/Hydration

- `mounted` 机制保持不变——避免 hydration mismatch
- 依赖 localStorage 的内容必须在客户端渲染

### 5. 功能层不做减法

- 排序维度、筛选维度、对比指标等核心功能**保留**
- 可以改交互方式（收纳、折叠、重新分组），但不删除用户可触达的功能维度
- 如果确实要删除某个功能，需要明确理由，不能仅凭"50% 以上用户不用"的猜测

# 第二部分：设计原则

以下是方向性指导。改动时对照思考，但不是逐条执行的 checklist。

### 产品定位：Which city?

产品定位就是域名本身——**whichcity.run**。用户带着一个问题来："去哪座城市？"

这个问题可能具体化为：
- 我应该搬到哪个城市？
- 这两个 offer 对应的城市，哪个生活更好？
- 作为游牧族，下一站去哪？
- 我好奇某个城市到底怎么样？

场景不同，但核心问题相同：**which city**。产品要做的是帮用户回答这个问题——用数据、用对比、用排名。

不是城市数据库（不做 API 平台），不是探索型社区（没有内容运营资源）。

### 信息层级：有序呈现，而非信息删减

当前问题不是"数据太多"，而是"数据没有层级"——30+ 字段以相同视觉权重平铺。
改进方向是**建立层级**，而非删除数据：
- 重要的数字更大、更醒目
- 次要的数字更小、更安静
- 补充信息可以折叠，但"补充"和"核心"的界限取决于产品定位（见上条）

### 三秒规则

用户在任何页面，3 秒内应该能回答"我在看什么"。这不需要给出结论，但需要让人知道这里有什么、怎么用。

### 个性化数据

用户选择了职业和收入水平后，数据应该围绕这个身份展示。不给"全局视角"——没人需要同时看 26 个职业的数据。

### 展示层做减法的方式

当我们想降低某个信息的视觉权重时，优先用以下手段（按优先级排序）：
1. **缩小视觉权重**（字号、颜色、位置）
2. **收纳到面板/下拉**（保留功能，减少占用面积）
3. **折叠**（默认隐藏，点击展开）
4. **移至其他页面**（如从详情页移到方法论页）
5. **删除**（最后手段，需要明确理由）

注意：我们没有用户使用数据，无法量化"X% 用户用这个功能"。**不要凭直觉声称某功能"只有 1% 用户用"然后删掉它**。如果不确定，保留但降低视觉权重。

# 第三部分：参考方案

> **以下所有内容仅供参考**，是 redesign 讨论中产生的想法和素材。
> 实施时应结合实际情况（用户反馈、技术可行性、工作量）选择性采纳。
> 不要把这些当作规范逐条执行。

---

## A. 身份系统参考

### 身份三要素

| 要素 | UI 呈现 | 存储 | 默认值 |
|------|---------|------|--------|
| 职业 | 下拉选择，26 个职业 | localStorage | 软件工程师 |
| 收入水平 | 自然语言标签（见下表） | localStorage | 中位数 |
| 预算偏好 | moderate / budget | localStorage | moderate |

### 收入水平标签（替代原始倍率数字）

薪资倍率逻辑保留（跨城市收入换算的唯一合理机制），但 UI 标签可以更直觉：

| 标签 | 对应倍率 | 含义 |
|------|---------|------|
| 偏低 | ×0.7 | 初级 / 低成本地区 / 非核心岗位 |
| 中位数 | ×1.0 | 默认值 |
| 偏高 | ×1.5 | 资深 / 高绩效 / 核心团队 |
| 很高 | ×2.0 | Staff+ / 管理层 / 顶级公司 |
| 极高 | ×3.0 | Principal / C-level / FAANG 天花板 |

### 降级方案

大量用户不从首页进入（Google 搜索、社交分享、直接链接）。如果 localStorage 没有已保存身份，子页面可以：
- 顶部出现一次性紧凑提示条："当前显示 XXX 职业 · 中位数收入 · [选择你的职业]"
- 点击展开设置，设好后消失
- 不阻断渲染，不遮挡主内容

## B. 首页参考方案

当前首页：搜索框 + 6 个硬编码热门城市。信息密度低。

一种思路——"决策起点"：

| 模块 | 内容 | 目的 |
|------|------|------|
| 身份选择区 | 职业 + 收入水平 + 预算偏好 + 可选"你在哪座城市？" | 建立个性化上下文 |
| 搜索框 | 保留，但不做视觉中心 | 已有目标城市的用户直接跳转 |
| 为你推荐的 Top 5 | 按身份计算储蓄率最高 5 城 | 用户立刻看到产品价值 |
| 排行榜入口 | 文字链接 | 不抢首屏空间 |

技术约束：Top 5 依赖 localStorage 身份，必须客户端计算。可用精简 JSON 或预计算常见组合。

## C. 城市详情页参考方案

当前问题：30+ 数据点以相同视觉权重平铺。

一种思路——建立信息层级（不是折叠一切，是区分主次）：

**首屏区域**：城市名 + 一句结论性描述 + 少量核心数字（如储蓄、安全、生活压力）+ 对比入口

**主体区域**（默认展开，通过视觉权重区分主次）：
- 财务组：收入、支出、储蓄（大号）→ 税率详情、储蓄率（小号）
- 住房组：房价、买房年数、月租（大号）→ 排名对比（小号）
- 工作组：工时、时薪、假期
- 环境组：AQI、网速、直航 → 气候详情、月度图表

**补充区域**（可折叠）：数字游民、相似城市详细差异

为什么第二层不折叠：城市详情页是 SEO 主要流量入口，核心数据保持可见。

## D. 排行榜参考方案

当前 22+ Tab 系统基本可用，主要改进方向：

- Multi 复合排序模式可考虑移除（需要用户理解加权概念）
- 子指标独立排序可考虑移至 Methodology 页
- 气候筛选 UI 可从多行 pill 收纳为下拉面板
- 表格可考虑默认 30 行 + 加载更多

---

## E. 对比页参考方案

- 默认展示可精简为核心指标（财务 + 住房 + 4 个指数 + 气候）
- 工作、环境、Nomad 可放入展开区域
- Win-count 可改为按组统计（"财务面 A 胜、生活面 B 胜"），比单一总数更有参考价值
- 保留：城市切换下拉、并排气候图

---

## F. NavBar 参考方案

一种思路——NavBar 瘦身：
- 导航链接：首页、排行、方法论
- 偏好齿轮：只控制语言、货币、主题（低频设置）
- 职业/预算/收入模式移至首页或详情页（场景化嵌入）

---

## G. 删除 / 弱化候选清单

以下是讨论中提出的候选项，**不是最终决策**：

### 可考虑删除

| 内容 | 考虑理由 |
|------|---------|
| Multi 复合排序模式 | 需要理解加权排名概念，门槛高 |
| 排行榜子指标独立排序 | 更像数据验证功能而非用户功能 |
| LINE 和 Reddit 分享 | 可能转化率低（无数据验证） |
| 排行榜 Tab emoji 分组前缀 | 更像代码结构可视化 |

### 可考虑弱化

| 内容 | 方向 |
|------|------|
| 收入模式切换 | 从全局设置移至详情页财务区域 |
| 数字游民信息 | 从独立大区块改为可展开区域 |
| 气候月度图表 | 移入环境组内部 |
| 导航栏多色按钮 | 可考虑统一色调 |
| 设置面板 | 减少同时暴露的选项数量 |
| 相似城市差异列表 | 简化为关键差异标签 |

---

## H. 实施方向（非优先级排序）

以下是可能的改进方向，具体做不做、先做哪个，视实际情况决定：

- **首页增加信息密度**：让新用户立刻感知到产品价值
- **详情页建立信息层级**：视觉主次分明，减少认知负担
- **设置场景化分布**：把设置项放到用户需要它的地方，而不是全塞在 NavBar
- **排行榜精简入口**：降低上手复杂度

## I. 改动前检查清单

- [ ] 现有 URL 会 404 吗？（不允许）

- [ ] SEO 核心内容仍然默认可见吗？
- [ ] 从非首页入口进来的用户体验如何？
- [ ] 是在改展示方式，还是在删功能？（后者需要明确理由）
