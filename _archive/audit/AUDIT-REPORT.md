## 执行摘要

本次审计对 WhichCity 项目的全部数据资产（154座城市、81国税务数据、汇率系统、复合指数、气候数据、i18n翻译）进行了5轮独立审查，共执行3个自动化审计脚本、4个并行人工审查流程。

### 关键发现概览

| 严重等级 | 发现数 | 说明 |
|---------|--------|------|
| 🔴 **致命** (P0) | **8** | 数据计算错误、数据不可追溯、系统性偏差 |
| 🟠 **严重** (P1) | **14** | 数据质量严重缺陷、逻辑漏洞、误导用户风险 |
| 🟡 **中等** (P2) | **23** | 数据精度不足、文档过期、设计局限 |
| 🟢 **轻微** (P3) | **12** | 最佳实践偏差、小范围数据缺失 |
| **合计** | **57** | — |

### 一句话定性

> **这个项目在"看起来专业"和"真正专业"之间存在一条危险的鸿沟。** 表面上数据覆盖广泛、系统完整，但深入挖掘后发现：新增20城的数据质量远低于老城、复合指数存在系统性计算错误、收入数据定义混乱到连项目自己都分不清是"税前"还是"税后"、气候月降雨量能跟年降雨量差出50%。如果用户依据这些数据做出人生重大决策（如移民、求职），本审计团队对此深表担忧。

## 第二章 数据可信性深度审计

### 2.1 🔴 收入数据——系统性混乱（P0级致命问题）

#### 事实发现

`averageIncome` 与 `professions` 中位数的关系极不一致:

| 城市 | averageIncome | 职业中位数 | 比率 | 问题 |
|------|--------------|-----------|------|------|
| 斯普利特(146) | $24,000 | $2,000 | **12.0x** | 荒谬 |
| 弗洛里亚诺波利斯(152) | $12,000 | $1,500 | **8.0x** | 荒谬 |
| 瓦伦西亚(144) | $30,000 | $4,000 | **7.5x** | 荒谬 |
| 蒙得维的亚(148) | $14,400 | $2,000 | **7.2x** | 荒谬 |
| 巴亚尔塔港(158) | $14,000 | $2,000 | **7.0x** | 荒谬 |
| 拉斯帕尔马斯(149) | $26,000 | $4,000 | **6.5x** | 严重 |
| 波尔图(143) | $22,000 | $3,500 | **6.3x** | 严重 |
| 班斯科(145) | $12,000 | $2,000 | **6.0x** | 严重 |
| 暹粒(155) | $3,000 | $500 | **6.0x** | 严重 |
| 坎昆(157) | $11,000 | $2,000 | **5.5x** | 严重 |
| 东京(3) | $23,349 | $55,000 | **0.42x** | 反向严重 |
| 大阪(106) | $20,343 | $50,000 | **0.41x** | 反向严重 |
| 京都(159) | $20,011 | $49,100 | **0.41x** | 反向严重 |
| 横滨(139) | $22,442 | $52,000 | **0.43x** | 反向严重 |

**审计结论**: 

1. **新增的20座城市（ID 140-159）的 `professions` 数据明显不是USD**。 斯普利特的 `averageIncome` 是$24,000（与克罗地亚实际收入大致吻合），但其职业薪资中位数仅$2,000。这极度暗示：**这些城市的 `professions` 值单位是月薪（本地货币）而非年薪（USD）**。
2. **日本城市的 `averageIncome` 疑似使用的是"税后"收入**，而 `professions` 是税前。在 `taxData.ts` 中日本并未标记 `dataIsLikelyNet=true`，但中国/泰国/越南/印度标记了。
3. **这意味着整个项目的收入数据存在根本性的"计量单位混乱"** — 有些城市是年薪USD、有些是月薪本地货币、有些是税后、有些是税前。当这些数据被放在一起排名和比较时，结论是完全不可信的。

#### 三重验证

- ✅ 脚本验证: 比率分析检测出24个异常
- ✅ 外部知识验证: 克罗地亚平均工资约€1,500/月，斯普利特$24,000/年合理，但项目里的$2,000职业值明显折射月薪
- ✅ 模式分析: 所有20座新城市（140-159）无一例外都有这个问题

### 2.2 🔴 生活成本比率——系统性违规（P0级）

项目文档声称 `costBudget` 应为 `costModerate` 的60-75%。

**自动化脚本检测结果**: **135/154 (87.7%) 的城市违反此规则**

典型违规:

| 城市 | costModerate | costBudget | 比率 |
|------|------------|----------|------|
| 重庆 | $1,100 | $429 | 39% |
| 成都 | $1,200 | $468 | 39% |
| 广州 | $1,600 | $656 | 41% |
| 北京 | $2,200 | $924 | 42% |
| 上海 | $2,600 | $1,170 | 45% |
| 东京 | $3,200 | $1,440 | 45% |

**审计结论**: 要么文档规则是错的，要么135座城市的数据都是错的。对于一个自己定义的规则自己都不执行的项目，这说明了**数据录入阶段没有任何验证机制。**

### 2.3 🟠 薪资复制粘贴（P1级）

| 城市组 | 共享完全相同的26个职业薪资 |
|--------|------------------------|
| 普吉岛(147)、帕岸岛(153)、苏梅岛(154) | ✓ 完全相同 |

三个泰国海岛城市共享完全相同的薪资矩阵。虽然它们确实在同一国家，但这是三个经济特征完全不同的地方（普吉是旅游中心，帕岸以Full Moon Party闻名，苏梅是高端度假地）。**这是典型的数据造假偷懒。**

### 2.4 🟠 同国城市数据"模板化"（P1级）

自动化检测发现 **23个同国相同值模式**:

| 国家 | 城市数 | 相同字段 | 值 |
|------|--------|---------|---|
| 美国 | 20城 | annualWorkHours | 1,811（所有城市完全一样） |
| 美国 | 20城 | paidLeaveDays | 0（所有城市完全一样） |
| 中国 | 7城 | annualWorkHours | 2,170 |
| 中国 | 7城 | paidLeaveDays | 5 |
| 日本 | 6城 | annualWorkHours | 1,607 |
| 泰国 | 5城 | annualWorkHours/paidLeaveDays | 相同 |
| 墨西哥 | 5城 | 同上 | 相同 |
| ...及更多 | | | |

**审计结论**: 工时和假期数据显然是**国家级数据被无差别复制到所有城市**。虽然对于某些指标（如法定假期天数）国家级数据是合理的，但年工时在同一国家的不同城市间差异可以很大（如中国的 "996" 主要集中在科技行业密集城市，并非全国一致）。**项目应明确标注哪些是国家级数据、哪些是城市级数据。**

### 2.5 🟡 住房价格可疑值（P2级）

| 城市 | housePrice (USD/m²) | 常识判断 |
|------|---------------------|---------|
| 芝加哥 | $1,805 | 偏低，作为美国第三大城市实际约$3,000-5,000 |

### 2.6 🟡 气候数据月降雨量不一致（P2级）

**30/154 (19.5%) 的城市月降雨量总和与年降雨量偏差超过15%**

最严重案例:

| 城市 | 月度总和 | 年降雨量 | 偏差 |
|------|---------|---------|------|
| 台北 | 1,835mm | 2,405mm | -24% |
| 拉各斯 | 1,804mm | 1,425mm | +27% |
| 麦德林 | 1,229mm | 1,685mm | -27% |
| 深圳 | 1,605mm | 1,933mm | -17% |

## 第三章 复合指数验证

### 3.1 🔴 安全指数——新城市系统性计算错误（P0级）

**脚本复算结果**: 20/154城市的安全指数与子指标加权平均值不符（偏差>0.5分）

其中**16座是新增城市（ID 140-159）**，偏差最大的:

| 城市 | 存储值 | 复算值 | 偏差 |
|------|--------|-------|------|
| 巴亚尔塔港(158) | 39 | 52.4 | **-13.4** |
| 库斯科(156) | 53 | 65.9 | **-12.9** |
| 弗洛里亚诺波利斯(152) | 43 | 53.6 | **-10.6** |
| 拉斯帕尔马斯(149) | 73 | 83.0 | **-10.0** |
| 普拉亚德尔卡门(142) | 39 | 48.2 | **-9.2** |
| 蒙得维的亚(148) | 57 | 66.1 | **-9.1** |

**审计结论**: 新增的20座城市在使用 `add-safety-v2.mjs` 脚本时明显有问题 — 存储的 `safetyIndex` 与按照文档声称的 35/30/20/15 权重算出来的值**系统性偏低**。说明要么：
1. 使用了不同的权重公式
2. 子指标在入库时被修改了但指数没重算
3. 完全是手工录入的"大概值"

这直接摧毁了项目声称的"透明可追溯"的方法论基础。

|--------|-----------|------|
| 安全指数置信度 | 3 | 子指标个数与置信度标签不匹配 |

## 第四章 税务引擎审计

### 4.1 概述

- 覆盖国家: 81国 ✅ 与声称一致

### 4.2 🔴 硬编码汇率 vs 动态汇率——双轨系统（P0级）

**这是架构层面的致命缺陷。**

`taxData.ts` 中每个国家都硬编码了一个 `usdToLocal` 汇率：

| 货币 | taxData硬编码 | exchange-rates.json | 偏差 |
|------|--------------|-------------------|------|
| EUR | 0.92 | 0.8564 | **-7.2%** |
| JPY | 150 | 158.53 | **-5.4%** |
| CNY | 7.24 | 6.84 | **-5.5%** |
| INR | 83.5 | 92.57 | **+9.8%** |
| SGD | 1.34 | 1.27 | **-4.9%** |

**代码分析**: `computeNetIncome()` 会**优先使用动态汇率**（如果传入），否则回退到硬编码值。但问题是：
1. 这两组汇率**同时存在于代码中**，没有任何同步机制
2. 如果客户端没有传入 `dailyRates`，所有计算将使用过时的硬编码值
3. 硬编码汇率的 EUR 偏差 7.2%，影响13个欧元区国家的**所有税后收入计算**

### 4.3 🟠 "dataIsLikelyNet" 静默跳过（P1级）

4个国家标记了 `dataIsLikelyNet=true`: 中国、泰国、越南、印度

对于这些国家，`computeNetIncome()` 直接返回税前值并声称有效税率为0%:

```javascript
if (tax.dataIsLikelyNet) {
  return { netUSD: grossUSD, effectiveRate: 0, confidence: "low" };
}
```

**问题**: 用户选择"税后收入"模式时，这些国家显示**0%税率**，这是极其误导人的。中国个人所得税最高45%、印度30%。项目应该显示"数据可能已为税后"的警告，而不是假装税率为零。

### 4.4 🟡 美国州税验证（P2级）

全部13个美国城市的州税覆盖已验证正确:
- 纽约: 州税+市税(4-10.9%) ✓
- 加州5城: 1-13.3% ✓
- 零税州 (FL, WA, TX, NV): 0% ✓
- 其他州: 税率与2025实际一致 ✓

### 4.5 🟢 瑞士州级税率过度简化（P3级）

苏黎世和日内瓦使用了单一平税率（26%和34%），而实际上两个州都有累进式州税系统。简化是可以接受的近似，但应注明。

### 5.1 🟠 无时间戳——数据陈旧不可知（P1级）

- ✅ 检查API Key是否存在

- ❌ 不处理JSON解析失败
- ❌ 不记录更新时间
- ❌ 没有陈旧数据告警

客户端货币转换有安全的回退（缺失货币默认1:1 USD比率），不会导致NaN或崩溃。

## 第六章 数据架构与使用方式审计

### 6.1 🔴 "相似城市"算法空值处理缺陷（P0级）

21维欧几里得距离算法中，空值使用了 `?? 0` 替代:

```javascript
c.housePrice ?? 0,        // null → 0
c.monthlyRent ?? 0,       // null → 0  
c.airQuality ?? 0,        // null → 0（但AQI 0 = 最佳空气！）
c.internetSpeedMbps ?? 0, // null → 0
c.annualWorkHours ?? 0,   // null → 0
```

**问题**:
1. `airQuality` 用0替代null意味着**缺失空气质量数据的城市被视为空气最好的城市**
2. 缺失住房数据的城市被视为"免费住房"
3. 缺失网速数据的城市被视为"没有互联网"

这导致**缺失数据的城市会被系统性地推荐为"与廉价城市相似"**。算法完全不可信。

### 6.2 🔴 Life Pressure 指数魔法数字（P0级）

`clientUtils.ts` 中的"购房年数"计算:

```javascript
const yearsToHome = (city.housePrice * 70) / income;
```

这个 `70` 是什么？70平方米？70%首付？完全没有文档说明。`999999` 也未解释。**一个核心指数的计算中有未解释的魔法数字，这是数据项目的大忌。**

### 6.3 🟠 数据加载无验证（P1级）

`dataLoader.ts` 使用 `readFileSync` + `JSON.parse`，没有:
- 没有try-catch
- 没有schema验证
- 没有类型检查
- 文件损坏或格式错误 = 整个服务器崩溃

### 6.4 🟠 排名中的null处理（P1级）

排名使用 `nullLast()` 将null放在最后，这是合理的。但在**多选复合评分**模式中，如果某一维度所有值都是null：
- `Math.min(...[])` = `Infinity`
- 归一化过程可能产生 `NaN`

虽然有过滤器保护，但链条脆弱。

### 6.5 🟡 气候降雨阈值硬编码（P2级）

排名页面的气候过滤器中，降雨量阈值 [600mm, 1200mm] 是硬编码的，而温度等其他维度使用动态百分位。用户体验不一致。

### 6.6 🟡 OG图片硬编码统计数字（P2级）

首页OG图片中硬编码了 "79 tax systems"，但实际已有81国。此类信息应从数据常量中动态获取。

## 评审团队终审意见与建议

### 总体评价

让我们直说吧。

这个项目**在外行人看来很不错** — 154个城市、26个职业、81国税务、4种语言，数据覆盖面确实令人印象深刻。UI大概也做得不错。但作为数据审计团队，我们的工作不是看"多宽"，而是看"多深"。

**往深看，这个项目的数据基础存在结构性问题：**

1. **最核心的收入数据连"是什么"都说不清楚。** 有些是年薪、有些像月薪；有些是税前、有些标记了"可能是税后"但标记不完整。当一个城市比较工具连"比较的到底是什么"都不确定时，它提供的任何"洞见"都是建立在流沙之上的。

2. **新增的20座城市是在"催产"。** 它们的数据质量显著低于老城数据 — averageIncome与professions脱节、安全指数计算错误、薪资复制粘贴。很明显这批城市在数据准备不充分的情况下被匆匆入库。

3. **自己制定的规则自己不遵守。** 文档说costBudget是costModerate的60-75%，实际87.7%的城市违规。这不叫"灵活"，这叫"规则是摆设"。

4. **税务引擎有一个隐藏的定时炸弹** — 硬编码汇率和动态汇率双轨并行，EUR偏差7.2%。每个用这个工具做薪资比较的欧洲用户，看到的税后收入**都是错的**。

5. **数据完全不可追溯。** 没有采集日期、没有来源链接、没有版本管理。项目无法回答："这个数据是什么时候的？"这对一个号称帮助用户做人生重大决策的工具来说，是不可接受的。

### 毒舌但诚恳的建议
