## 讨论记录

### 任务 1：排名类型调查

**🔧 [21:00] 工程师**: Boss 指出报告写的是 dense ranking，但实际表现像 standard ranking。我去查了代码——`ranks[i] = same ? ranks[i - 1] : i + 1`。当 i = 3（位置第4）且前两行同分时：1, 2, 2, 4。这就是 standard ranking，不是 dense ranking（dense 应该是 1, 2, 2, 3）。

**🧪 [21:01] 测试员**: 所以报告描述有误？代码本身没问题？

**🔧 [21:02] 工程师**: 对。代码行为就是 standard ranking。Boss 要求的也是 standard ranking。代码不需要改，报告描述当时写错了。

**🃏 [21:03] 捣蛋鬼**: 为什么上一次说是 dense？

**🔧 [21:04] 工程师**: 讨论阶段确实说的是 dense，但实现时 `i + 1` 的逻辑自然产生了 standard ranking。实现覆盖了设计意图，结果反而更好——Boss 也认同 standard 更直观。

**🔧 [21:05] 工程师 [决定]**: 无需代码变更。已确认现有行为是 standard ranking。

### 任务 2：排名精度审计

**🔧 [21:06] 工程师**: 逐项审计了 getVal（用于同分判定）和 renderCells（用于显示）的精度对应关系。

**🧪 [21:07] 测试员**: 我列出审计结果：
- 货币字段（income/savings/expense）：`computeNetIncome` 返回 `Math.round(netUSD)` 整数，`formatCurrency` 也是 `Math.round`，getVal 直接比较整数 → ✓
- 指标字段（lifePressure/safety/healthcare/freedom）：显示 `.toFixed(1)`，getVal 用 `Math.round(val * 10)` → 精度一致 ✓
- 小数字段（doctors/beds/lifeExpectancy/democracy）：显示 `fmtN(v, "", 1)` 即 `.toFixed(1)`，getVal 用 `Math.round(val * 10)` → ✓
- 整数字段（numbeo/homicide/gpi/gallup/press/cpi/uhc）：数据源就是整数（从 cities.json 确认），显示和比较都是原值 → ✓
- hourlyWage：`Math.round(hourlyWage * 100)`（2位精度），formatCurrency 内部也有 Math.round → ✓
- bigMacPower：`.toFixed(1)` 和 `Math.round(val * 10)` → ✓

**🔧 [21:08] 工程师**: 全部一致，无需修改。

**🔧 [21:09] 工程师 [决定]**: 精度已经对齐，无需代码变更。

### 任务 3：标签二三行间距

**🎨 [21:10] 设计师**: 当前标签容器用 `space-y-1.5`（6px gap），但第三行（指标行）额外有 `pt-1`（4px），导致二三行间距 = 6+4 = 10px，一二行间距 = 6px。

**🔧 [21:11] 工程师 [决定]**: 移除 `pt-1`。现在三行间距统一由 `space-y-1.5` 控制，全部 6px。

### 任务 4 & 5：统计描述重写

**🃏 [21:12] 捣蛋鬼**: Boss 说三个问题：(1) "16 项数据"包含了 4 个指标，再说"4 大指标"是自我重复；(2) "4 大指标"口气太大；(3) 没覆盖税、语言、气候等。

## 关键决策 / Key Decisions

1. **Ranking 类型**：代码已经是 standard ranking，报告描述有误→不改代码
2. **统计行**：从枚举精确数字改为描述覆盖领域，更谦逊自然不易过时
3. **命名权威**：城市详情页为权威命名来源，排行榜 tab 跟随一致
4. **热门城市**：6 座城市覆盖 5 大洲，编译时确定，零运行时开销
5. **浅色标签**：深一个 Tailwind 档次（50→100, 100→200），既可见又不过分
